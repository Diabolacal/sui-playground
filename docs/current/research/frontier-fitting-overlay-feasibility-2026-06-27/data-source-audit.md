# Data-Source Audit — Fitting Tool (2026-06-27)

**Status:** Active research artifact (Workstream B). Where the data for a fitting tool exists today,
what is missing, and the extraction route. Sources: a read-only shallow clone of `Diabolacal/EF-Map`
@ `b1cd69e` inspected directly + via Subagent B; example records verified first-hand against
`blueprint_data_v5.json`.

> **Headline:** **A meaningful slice of the catalog is already free** (module identity, mass, volume,
> category/group, and build recipes for all 16 Cycle 6 modules live in EF-Map's public
> `blueprint_data_v5.json`). The **classic per-type stats** (powergrid, HP, velocity, capacitor, warp,
> inertia) are **extractable** via EF-Map's existing Phobos/FSD pipeline. The **two hard gaps** are the
> **polyomino footprint shapes** and the **interior/exterior + hardpoint metadata** — these were not
> found in any static data and will almost certainly be **hand-authored** from the in-game UI.

## 3. Fitting-data availability matrix

Verdict key: **(a)** present in EF-Map data now · **(b)** absent from EF-Map but extractable via the
Phobos/FSD dogma pipeline · **(c)** not found anywhere → hand-author.

| Need | Verdict | Evidence / route |
|---|:--:|---|
| Module identity, mass, volume | **(a)** | `blueprint_data_v5.json` items (table above) |
| Module build recipe (shopping list) | **(a)** | `blueprint_data_v5.json` recipes + `blueprintIDs` |
| Module group/category (broad) | **(a)** | `groupName`/`categoryName` (UI labels need authoring) |
| Ship mass, volume | **(a/b)** | `shipData.ts` (13 ships) + `types.fsdbinary` |
| Ship specific heat, fuel capacity | **(a/b)** | `shipData.ts` + `typeDogma` |
| Ship HP, max velocity, warp speed | **(b)** | `extract_ships.py` targets `hullHp/maxVelocity/warpSpeedMultiplier` |
| Ship powergrid capacity (e.g. 15 MW) | **(b)** | `extract_ships.py` → `powerOutput`; dogma "Fitting" category |
| Ship capacitor capacity / recharge | **(b)** | standard dogma attrs (`capacitorCapacity`, recharge) |
| Ship inertia modifier (agility) | **(b)** | standard dogma attr (not explicitly named in script; open dump) |
| **Module powergrid usage (MW)** | **(b)** | per-module dogma (`powerLoad`/equiv); **not** in blueprint_data — needs an extraction run over module typeIDs |
| **Conductance** | **(c)** | new Cycle 6 stat; zero hits anywhere (20% chance it's an unnamed dogma attr) |
| **Fuel Impulse, Fuel Rate (ship)** | **(c/b)** | new Cycle 6 terms; not found by name (fuel-rate analog `burnRateInSec` seen only for stations) |
| **Module footprint SHAPE (polyomino)** | **(c)** | **highest-risk** — not in any of 120+ Phobos files / 2.1M records; ~**90%** not in static data |
| **Interior/exterior flag** | **(c)** | not found; new Cycle 6 mechanic |
| **Hardpoint provider/consumer type** | **(c)** | not found; new Cycle 6 mechanic (no hi/mid/low-slot analog) |

## 4. The footprint-shape verdict (the crux)
**The polyomino grid masks are not present in any extractable static data (≈90% confidence).** No
occurrence of "footprint/polyomino/grid mask/slot shape" in the Phobos catalog, dogma definitions, or
generators. The closest near-misses are explicitly *not* it: `collisions` (3D meshes), `graphicIDs.
sofLayout` (the cosmetic 3D "slot overlay fitting" for the ship model, not the gameplay grid),
`spaceComponentsByType` (deployable placement rules). EVE Online used **integer slot counts**, not
polyominoes — Cycle 6's grid is a **new mechanic** that almost certainly lives in the game **UI/engine
layer**, not in FSD data. The residual ~10%: a new, un-probed FSD domain, or unnamed dogma attributes.

**Consequence:** footprint shapes (and the interior/exterior + hardpoint metadata) must be
**hand-authored** from the in-game UI into the versioned catalog — exactly the pattern EF-Map already
uses for `shipData.ts`. This is ~16–30 small entries for Cycle 6; tractable, and it is the dominant
manual cost of a data-backed fitting tool. See [`fitting-model-and-rules.md`](fitting-model-and-rules.md) §5.

## 5. Candidate extraction route (recommended)
**Route A — extend EF-Map's existing Phobos/FSD extractor.** `tools/game-data-extractor/extract_ships.py`
already exists, already targets the right dogma attribute names, and the repo documents 87/87 FSD
loaders succeeding on the May 2026 client. **Extend it to iterate the 16 Module typeIDs (95302–95503,
+81846) and dump their full dogma attribute maps.** One run answers definitively: (a) which fitting
stats (powergrid usage, etc.) exist in dogma for Cycle 6 modules, and (b) whether `conductance` /
`inertia` / fuel-impulse appear as new attribute IDs. Prereqs: Python 3.12, the EVE Frontier client
installed, `code.ccp`/bin64 on path. Effort: **~2–4 hours** + post-processing to named attributes.

- **Route B (supplemental):** the live World API `typeDetails` (per `assembly.json`) — gives mass/
  volume/category only, **no** fitting stats. Useful as a name/icon source, not for stats.
- **Route C (mandatory for the gaps):** hand-author footprint shapes + interior/exterior + hardpoint
  metadata from screenshots into the catalog.
- **Route D (avoid):** VULTUR/eve-frontier-tools — universe data only, no ship/module data.

## 6. Confidence & what it means for the MVP
- **High confidence (build on it now):** module identity/mass/volume/recipe + ship base stats from
  `shipData.ts` → enough for the **palette, mass total, and build shopping list**.
- **Medium confidence (one extraction run):** module powergrid usage + ship powergrid/HP/velocity/
  capacitor → enough for the **power-management constraint** and most of the **stats pane**.
- **Author-required (no data):** footprint shapes, interior/exterior, hardpoints, and the new thermal/
  fuel-impulse stats → the **placement engine's geometry** and a few stats start **hand-authored +
  calibrated** (honest "estimated" flags), and harden as we test in-game.

**Net:** a **data-backed fitting MVP is feasible** — the geometry layer is the hand-authored part, and
that is bounded (16–30 entries) and exactly the kind of authoring EF-Map already does. The single
highest-leverage next data task is the **module-dogma extraction spike** (Route A); see
[`implementation-briefs/fitting-data-extraction-spike.md`](implementation-briefs/fitting-data-extraction-spike.md).
